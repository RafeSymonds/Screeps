# Cross-Tick State Boundaries

This document maps the highest-risk state boundaries in the Screeps AI runtime. Because the project uses a CPU-throttled planning model and multi-agent development, understanding where state persists and how it flows between subsystems is critical for avoiding regressions.

## 1. Task Lifecycle (Planning to Execution)

**Boundary**: `Memory.tasks` ↔ `TaskManager` ↔ `PlanManager` ↔ `CreepActions`

*   **Flow**:
    1.  `PlanManager` executes plans (throttled).
    2.  Plans create/update `TaskData` in the `TaskManager`.
    3.  `TaskManager` persists `TaskData` to `Memory.tasks`.
    4.  `TaskAssignment` links creeps to tasks via `taskId` in `CreepMemory`.
    5.  `CreepActions` executes logic based on the assigned task.
*   **Risks**:
    *   **Staleness**: If a plan is skipped, its tasks may become stale.
    *   **Dangling References**: If a task is pruned but a creep still has its `taskId`, the creep becomes "lost".
    *   **Inconsistent Rehydration**: Errors in `TaskCreation.ts` can cause tasks to vanish from `Memory` during the tick loop.
*   **Ownership**:
    *   **Core Logic**: `technical-architect` (`src/tasks/core/`)
    *   **Domain Implementation**: `economy-engineer` or `operations-engineer` (`src/tasks/definitions/`)

## 2. Spawn Demand (Tasks to Spawning)

**Boundary**: `Memory.tasks` ↔ `SpawnManager` ↔ `SpawnRequests`

*   **Flow**:
    1.  `SpawnManager` reads all active tasks via `TaskManager.getTasksForRoom()`.
    2.  Each task provides `TaskRequirements` (WORK/CARRY/MOVE/COMBAT parts needed).
    3.  `SpawnManager` calculates "Pressure" (Demand vs Supply) and generates `SpawnRequests`.
    4.  **Baseline Economy**: Requests generated by the `SpawnManager` itself use the `baseline:${role}:${roomName}` key prefix.
*   **Risks**:
    *   **Demand Collapse**: If a plan fails to create tasks, `SpawnManager` sees zero demand and stops spawning, leading to economic collapse.
    *   **Over-Spawning**: Incorrect `TaskRequirements` can cause the spawner to drain the energy budget on unnecessary creeps.
*   **Ownership**:
    *   **Heuristics & Demand**: `economy-engineer` (`src/spawner/`)
    *   **Integration**: `technical-architect`

## 3. Resource Reservations (Labor to Energy)

**Boundary**: `CreepMemory` ↔ `ResourceManager` ↔ `RoomEnergyState`

*   **Flow**:
    1.  When a creep is assigned to pick up energy, it writes `energyTargetId` or `remoteEnergyReserved` to its `CreepMemory`.
    2.  `ResourceManager` rehydrates these reservations at the start of the tick.
    3.  `TaskAssignment` and `CreepActions` check availability via `ResourceManager` before committing more labor.
*   **Risks**:
    *   **Ghost Reservations**: If a creep dies or its memory is cleared without updating the `ResourceManager`, energy may be "locked" and unavailable to other creeps.
    *   **Race Conditions**: If multiple tasks try to reserve the same energy source in a single tick without using the `ResourceManager` correctly.
*   **Ownership**:
    *   **Core Logic**: `technical-architect` (`src/rooms/ResourceManager.ts`)

## 4. Room Intelligence (Scouting to Strategy)

**Boundary**: `RoomMemory.intel` ↔ `RoomMemory.topology` ↔ `PlanManager`

*   **Flow**:
    1.  `ScoutingPlan` or `ScoutTask` updates `RoomIntel` and `RoomTopology`.
    2.  Strategic plans (`RemoteMiningPlan`, `ExpansionPlan`, `AttackPlan`) read this data to choose targets.
*   **Risks**:
    *   **Suicide Missions**: If `intel.lastHostileSeen` is stale, a plan might send vulnerable haulers into a room that was recently overtaken.
    *   **Pathing Failures**: Stale topology can cause `InterRoomRouter` to suggest impossible routes.
*   **Ownership**:
    *   **Data Structure**: `technical-architect` (`src/main.ts`)
    *   **Strategy & Updates**: `operations-engineer` (`src/rooms/`, `src/plans/definitions/`)

## 4. Resource Allocation (Economy to Support)

**Boundary**: `RoomMemory.supportRequest` ↔ `SupportPlan` ↔ `ResourceManager`

*   **Flow**:
    1.  A room in distress (e.g., newly claimed) writes a `RoomSupportRequest` to its memory.
    2.  `SupportPlan` (running in a surplus room) reads these requests and creates `SupportTasks`.
*   **Risks**:
    *   **Deadlocks**: If multiple rooms request support and the surplus room cannot prioritize, it may oscillate or fail to send help where needed most.
*   **Ownership**:
    *   **Economy Flows**: `economy-engineer`

## 5. Scheduling (CPU Budget to Plans)

**Boundary**: `Memory.planRuns` ↔ `PlanScheduler` ↔ `PlanManager`

*   **Flow**:
    1.  `PlanManager` checks `shouldRunPlan()` for each strategic pass.
    2.  `PlanScheduler` checks `Game.cpu.bucket` and `Memory.planRuns[key]`.
*   **Risks**:
    *   **Starvation**: If a critical plan (e.g., `DefensePlan`) is incorrectly marked with a high interval or low priority, it may be skipped when the bucket is low, leading to room loss.
*   **Ownership**:
    *   **Core Scheduling**: `technical-architect` (`src/plans/core/`)

## 6. Spawn Stats & Pressure (History-Aware Demand)

**Boundary**: `RoomMemory.spawnStats` ↔ `SpawnManager`

*   **Flow**:
    1.  `SpawnManager` calculates instant pressure based on demand vs supply.
    2.  Pressure is smoothed into an Exponential Moving Average (EMA) using `PRESSURE_ALPHA`.
    3.  Stats are persisted in `RoomMemory.spawnStats`.
*   **Risks**:
    *   **Echoing Demand**: If `PRESSURE_ALPHA` is too high, the spawner may over-react to a single tick of demand. If too low, it may fail to replace dying creeps in time.
    *   **Stale Stats**: If `SpawnManager` skips a tick (though it shouldn't), the EMA becomes slightly inaccurate.
*   **Ownership**:
    *   **Heuristics**: `economy-engineer`

## 7. Onboarding & Support signaling (Inter-Plan Priority)

**Boundary**: `RoomMemory.onboarding` / `RoomMemory.supportRequest` ↔ `SpawnManager`

*   **Flow**:
    1.  `SupportPlan` or `ExpansionPlan` writes flags to a room's memory.
    2.  `SpawnManager` reads these flags and applies `rolePriorityBoost()` to baseline economy requests.
*   **Risks**:
    *   **Priority Bloat**: If multiple plans apply boosts without coordination, the energy budget may be drained on low-priority "support" workers while the main room starves for miners.
*   **Ownership**:
    *   **Cross-Plan Signaling**: `technical-architect`

## 8. Long-Interval Persistence

**Boundary**: `GrowthPlan`, `ExpansionPlan`, `BasePlan`, etc. (Intervals > 25)

*   **Flow**:
    1.  These plans run rarely (every 25-50 ticks).
    2.  Their results (`RoomMemory.growth`, `RoomMemory.basePlan`) are treated as "Current Truth" by all other systems until the next run.
*   **Risks**:
    *   **Laggy Response**: If the environment changes (e.g., a room is lost), an `ExpansionPlan` running on a 50-tick interval may continue to request claimers for a target that is no longer viable.
*   **Ownership**:
    *   **High-Level Strategy**: `operations-engineer` or `base-specialist`
